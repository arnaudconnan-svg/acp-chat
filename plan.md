Leçons tirées depuis le code (sans hypothèses)

1. Le déclenchement Critic n’est pas piloté par needsSoberReadjustment.
- Le Critic se déclenche sur un autre ensemble de conditions: N1/recall forcé, dépassement longueur, risque procedural, tutoiement/vouvoiement, agency injection, violations théoriques, dans [server.js](server.js#L4664).
- needsSoberReadjustment est une décision de posture séparée dans [lib/pipeline.js](lib/pipeline.js#L295).
- Donc les cas “réajustement sobre sans réécriture Critic” et “réécriture Critic sans réajustement sobre” sont possibles par construction.

2. Le debug expose déjà des données Critic, mais le motif précis reste peu lisible en UI.
- Le backend renvoie criticTriggered, criticIssues, criticOriginalReply dans [lib/debugmeta.js](lib/debugmeta.js#L126).
- L’UI affiche surtout un résumé comptable (“corrigé N point(s)”) dans [public/index.html](public/index.html#L6457) et [public/admin.html](public/admin.html#L1960).
- La réponse initiale est visible, mais les raisons de déclenchement (ex: formalAddressRisk, contractLengthExceeded, humanFieldRisk) ne sont pas exposées dans les sections debug.

3. La mémoire peut conserver des éléments “peu utiles” car il n’existe pas de filtre déterministe de valeur clinique/produit.
- updateMemory et finalizeMemoryCandidate reposent sur LLM dans [lib/memory.js](lib/memory.js#L79).
- sanitizeMemoryItems filtre surtout les contenus théoriques/interprétatifs interdits, pas les items vagues ou triviaux dans [lib/memory.js](lib/memory.js#L5).
- Donc une ligne du type “partage des aspects personnels” peut passer même si elle est perçue comme peu utile.

Axes d’amélioration concrets imaginables depuis le code

1. Rendre le Critic audit-able dans le debug produit.
- Ajouter dans debugMeta une liste de triggerReasons déterministes (ex: contractLengthExceeded, formalAddressRisk, humanFieldRisk, agencyInjectionRisk, theoreticalViolationRisk) et l’afficher en français dans les sections Lecture interne de [public/index.html](public/index.html) et [public/admin.html](public/admin.html).

2. Clarifier la séparation “réajustement sobre” vs “Critic pass”.
- Afficher explicitement en debug que ce sont deux mécanismes différents avec leurs conditions propres, pour éviter l’impression de comportement arbitraire.

3. Filtrer la mémoire “faible valeur” avant persistance.
- Ajouter un post-filtre déterministe simple sur Contexte stable dans [lib/memory.js](lib/memory.js) pour rejeter les formulations génériques peu actionnables, avant écriture finale.
